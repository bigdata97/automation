Google Cloud Composer - Architecture and Working Mechanism

Google Cloud Composer is a fully managed workflow orchestration service built on Apache Airflow, allowing you to author, schedule, and monitor pipelines across clouds and on-premises environments.

üéØ What is Cloud Composer?

  • Managed Apache Airflow: Google provisions and manages Airflow components.
  • Serverless: No need to provision VMs manually.
  • Integrated: Works seamlessly with GCP services like BigQuery, Dataflow, GCS, Pub/Sub, etc.
  • Versioned: Composer environments are tied to specific Airflow versions.

üèóÔ∏è High-Level Architecture

Cloud Composer consists of:

  1. GKE Cluster: Underlying Kubernetes cluster where Airflow components run.
  2. Airflow Components:
    • Web Server: Airflow UI.
    • Scheduler: Triggers tasks based on DAG schedules.
    • Workers: Execute Python tasks.
    • Database: Cloud SQL for metadata storage.
    • Redis/Celery: For task queuing in CeleryExecutor mode.
  3. Cloud Storage Bucket: Stores DAGs, plugins, logs, and data dependencies.
  4. Cloud SQL: Relational DB to track Airflow metadata (DAG runs, task states).
  5. Google Cloud IAM: Manages access control to Composer environment resources.
  6. Logging & Monitoring: Stackdriver (Cloud Logging) collects logs; metrics go to Cloud Monitoring.

üß© How It Works - Execution Flow

  1. Author DAGs: Write Python DAG files and upload to the Composer bucket (gs://.../dags).
  2. Airflow Scheduler: Scans DAG directory and schedules tasks based on triggers.
  3. Task Queuing: Tasks are queued using Celery or Kubernetes Executors.
  4. Task Execution: Worker pods pick up tasks and execute them.
  5. Metadata & Logs: Stored in Cloud SQL and streamed to Cloud Logging + GCS.
  6. Monitoring: View DAG/task status in Airflow UI or Cloud Monitoring.
